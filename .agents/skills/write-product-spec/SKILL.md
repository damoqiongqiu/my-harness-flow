# write-product-spec

为此仓库中的重要功能编写产品 spec。

## 概览

此 SKILL 是本仓库的本地共享产品 spec 工作流。本地包装器和 workflow 直接将其作为规范的产品 spec 合约来依赖。

产品 spec 应使期望行为足够清晰，以至于 agent 可以正确实现它并在做变更时避免回归。专注于产品行为、UX、不变量和验证，而非实现细节。

将 spec 写入 `specs/` 下的适合仓库的路径。

如果仓库特定的包装 SKILL 或明确的 prompt 提供了精确的输出路径，遵循该路径。否则优先使用清晰的结构，如：

- `specs/<topic>/product.md`

## 编写前准备

收集编写 spec 所需的最少上下文：

**0. 先读目录规范**：检查 `specs/README.md` 是否存在，若存在则读取，了解本项目 spec 目录结构、命名规则和生命周期要求。

**1. 整合用户已有输入**：回顾当前会话中用户已提供的所有需求描述、文档、讨论内容（包括 prompt、issue 描述、附件材料等），作为 spec 内容的核心来源。**不要忽略用户输入自行创作**——spec 应忠实反映用户意图，而非 Agent 的猜想。

**常规准备：**

- 功能摘要
- 目标用户或 workflow
- 关键面向用户的行为和约束
- 已知边界情况
- 预期的验证计划
- 当周边 workflow 需要时的跟踪器或 issue 标识符

当请求遗漏了重要细节时，使用 `ask_user_question` 收集缺失的上下文，而非猜测。**但如果处于自动执行流程（由 `start-task` §3.1 调用），不要提问——将未解决的歧义记录到 §9 "待解决问题"中，做合理假设继续书写。**

如果功能包含 UI 或交互变更，询问是否存在 Figma mock。

### Figma 指引

如果存在 Figma mock，在 spec 中包含其链接。

如果用户未提供且会话是交互式的，询问是否存在 mock。否则，记录缺失并继续。不要默默忽略设计上下文。

例如，包含类似以下的简短说明：

- `Figma: <link>`
- `Figma: none provided`

无 Figma mock 是可以接受的，但缺失应被明确记录。

## 要写什么

使用以下结构：

### 1. 摘要

用几句话描述功能并陈述期望的结果。

### 2. 问题

解释正在解决什么用户或产品问题。

### 3. 目标

列出此变更必须达成的结果。

### 4. 非目标

列出明确排除在范围外的相关想法或后续事项。

### 5. Figma / 设计引用

如果存在，链接 Figma mock；否则明确说明未提供。

### 6. 用户体验

用具体、详尽、可测试的术语描述期望行为。目标是提供一份完整的用户可见行为的文本描述，审查者可通过测试、截图、视频和代码审查来验证。明确描述：

- 默认行为
- 状态转换
- 边界情况
- 空状态
- 错误状态
- 相关的键盘或交互期望

在有用时，将此节写为一组不变量或行为规则而非宽泛的散文。宁可包含太多相关细节，也不要使用模糊的摘要语言。

### 7. 成功标准

详细定义功能正确工作时应满足的条件。每个标准应映射到可观察的用户行为，且足够具体以便实现者或审查者可通过测试和检查代码来验证。偏好具体的、可观察的结果而非模糊的质量声明，并包含对正确性重要的关键状态、转换和边界情况。

### 8. 验证

描述应如何验证行为。偏好能清晰映射到测试、视频、截图或手动验证步骤的检查。

### 9. 待解决问题

明确指出未解决的产品决策，而非将其埋藏在叙述中。

## 编写指引

- 偏好具体行为而非期望性的措辞。
- 为实现者和审查者编写，而非为营销编写。
- 使 spec 足够精确以便 agent 可以遵循。
- 捕获不得回归的不变量。
- 包含易于在实现中遗漏的边界情况。
- 避免实现细节，除非对理解 UX 不可避免。

## 何时避免使用此 SKILL

当变更足够小以至于开销超过价值时，跳过产品 spec。粗略指南：spec 对重要功能最有用，而非小型编辑。

## 保持 spec 最新

如果实现改变了预期的产品行为，更新签入的产品 spec 使其仍与实际发布内容匹配。

已批准的 spec 可以在与代码相同的 PR 中实现。随着实现演进，在可能时将产品 spec、技术 spec 和代码的更新保持在同一 PR 中。

在以下情况变化时更新产品 spec：

- 面向用户的行为
- UX 细节
- 成功标准
- 验证期望

对于大型功能，实现者可视情况选择维护一份 `DECISIONS.md` 文件，汇总在 spec 和实现工作中做出的具体产品和技术决策。这是可选的，应在有助于保持工作一致时提供。
