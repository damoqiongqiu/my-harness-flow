# write-tech-spec

**目标项目路径规则**：spec 写入**目标项目**的 `specs/<topic>/tech.md`。如果当前工作目录在 harness framework，先确定目标项目路径再写入。

为此仓库中的重要功能编写技术 spec。

## 1. 概览

此 SKILL 是本仓库的本地共享技术 spec 工作流。本地包装器和 workflow 直接将其作为规范的技术 spec 合约来依赖。

技术 spec 应将产品意图转化为适合现有代码库的实现计划，记录架构选择，并使 agent 更容易执行和审查者更容易评估。

将 spec 写入 `specs/` 下的适合仓库的路径。

如果仓库特定的包装 SKILL 或明确的 prompt 提供了精确的输出路径，遵循该路径。否则优先使用清晰的结构，如：

- `specs/<topic>/tech.md`

## 2. 何时使用

在以下情况使用此 SKILL：

- 功能很重要
- 实现跨越多个模块或层次
- 有显著的架构决策或权衡
- 可扩展性很重要
- 审查者从在代码之前或与代码一起理解计划中获益

对于纯 UI 变更，技术 spec 可能不必要。务实行事。

## 3. 前置条件

最好先有产品 spec，以便技术计划锚定在已商定的行为上。

如果实现仍然过于不确定，先构建端到端原型然后根据学到的内容编写技术 spec 可能是更好的选择。当原型是减少歧义的最快方式时，不要强制编写推测性的技术 spec。

## 4. 编写前调研

起草技术 spec 之前：

1. 阅读相关的产品 spec（如果存在）。
2. 检查代码库中的已有模式。
3. 识别涉及的主要文件、类型、数据流和所属边界。
4. 理解当前行为及其不足之处。
5. 记录依赖项、部署约束、风险和可能的验证策略。

当可以直接检查代码时，不要猜测当前架构。

## 5. 要写什么

使用以下结构：

### 5.1 问题

陈述要解决的技术问题及其与产品行为的关系。

### 5.2 相关代码

指出最相关的文件、类型和入口点，以便实现可以从真实代码而非空白页开始。

例如：

- `src/module.py:42` — 用户流程的入口点
- `src/module.py (120-220)` — 可能会变化的状态和事件处理
- `src/components/button.tsx:10` — 可遵循的已有组件模式

### 5.3 当前状态

描述系统今天如何工作以及对此功能重要的局限性。

### 5.4 建议的变更

展开实现计划。明确描述：

- 哪些模块或组件会变化
- 将引入的新类型、API 或状态
- 数据流和事件流
- 所属边界
- 此设计如何遵循仓库中已有的模式

### 5.5 端到端流程

解释主要用户交互或系统行为的系统路径。

### 5.6 风险与缓解

指出可能的失败模式、回归、迁移关注点或部署风险。

### 5.7 测试和验证

列出为实现与期望行为匹配所需的测试和其他验证。

### 5.8 后续跟进

记录不应阻塞当前实现的延迟清理、扩展或未来工作。

## 6. 编写指引

- 将计划立足于真实的代码库结构和模式。
- 偏好具体的实现指南而非通用的架构语言。
- 解释为什么建议的设计适合此仓库。
- 当有多个合理路径时，明确指出权衡。
- 保持文档简洁，但足够具体以便 agent 可以从中实现。

## 7. 保持 spec 最新

如果实现偏离了计划的架构，更新签入的技术 spec 使其仍与现实匹配。

已批准的 spec 可以在与代码相同的 PR 中实现。随着实现演进，在可能时将产品 spec、技术 spec 和代码的更新保持在同一 PR 中。

在以下情况变化时更新技术 spec：

- 模块边界或所属关系
- 实现顺序
- 风险或缓解措施
- 验证策略
- 部署或依赖假设

对于大型功能，实现者可视情况选择维护一份 `DECISIONS.md` 文件，汇总在 spec 和实现工作中做出的具体产品和技术决策。这是可选的，应在有助于未来 agent 或审查者理解设计如何演进时提供。
